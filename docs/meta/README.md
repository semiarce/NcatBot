# 元文档

本文件供 AI Agent 参考，描述 NcatBot 文档的结构约定、内容规范、写作风格和维护流程。

---

## 一、目录编排

### 1.1 整体结构

```text
docs/
├── README.md          # 文档中心首页：导航表 + 完整目录树 + 详细目录
├── architecture.md    # 架构总览（不限 locs）
├── guide/             # 使用指南（面向用户）
├── reference/         # API 参考（面向开发者）
├── contributing/      # 贡献指南（面向贡献者）
└── meta/              # 文档规范（面向 AI Agent）
    └── README.md      # 本文件
```

### 1.2 README.md 的用途与结构

每个目录下的 `README.md` 充当该目录的**入口文档**，必须包含：

1. **一句话说明** — 本目录涵盖什么内容，面向哪类读者
2. **内容概览表格** — 文件名/子目录、说明、难度/篇数
3. **快速启动** — 给读者一个最直接的起步路径（1-3 步）
4. **导航指引** — 列出子文档并附一句注释，方便直接跳转

根目录 `docs/README.md` 额外包含：完整目录树（展示所有文件）、各子目录的详细清单。

### 1.3 文件命名约定

| 场景 | 命名格式 | 示例 |
|------|----------|------|
| 有明确阅读顺序的系列文档 | `数字前缀.主题.md` | `1.quick-start.md` |
| 同一主题拆分为上下篇 | `数字+字母.主题.md` | `3a.loading.md` / `3b.unloading.md` |
| 无强制顺序的参考文档 | `主题.md` | `segments.md` |
| 下划线风格（reference/ 惯例） | `数字_主题.md` | `1a_message_api.md` |

> **原则**：同一目录内风格保持一致；不要混用 `-` 与 `_`。

### 1.4 一级目录用途边界

| 目录 | 面向对象 | 内容侧重 |
|------|----------|----------|
| `guide/` | 使用插件的开发者 | 任务导向（"如何做 X"） |
| `reference/` | 查 API / 类型细节 | 参考导向（类、方法、参数列表） |
| `contributing/` | 框架贡献者 | 内部实现、设计决策、开发环境 |
| `architecture.md` | 所有人 | 系统全局视图、模块关系 |

## 二、内容规范

### 2.1 规模限制

- **普通文档文件**：目标 < 250 locs，硬限制 ≤ 400 locs（含空行）。
- **超限处理**：
  - 较大主题 → 在同一目录下拆分为 `Xa.主题.md` / `Xb.主题.md`
  - 非常大的主题 → 独立子目录，目录内放 `README.md` + 若干子文件
- **合并优先**：当两个相关文件合并后仍在 400 locs 以内时，优先合并为单文件以减少导航负担。
- **例外**：`architecture.md` 不受限制；`docs/README.md` 作为索引可适当超出。

### 2.2 渐进式披露

- **README.md / 快速入门** → 只覆盖最常用的 20% 功能
- **中级文档** → 覆盖常见进阶场景，不展开边缘情况
- **参考文档** → 完整列出所有 API，包括边缘情况和注意事项

不要在快速入门文档里出现复杂配置；不要在参考文档里花大量篇幅写教程。

### 2.3 每类文档要包含的内容

**指南文档（guide/）**
- 明确的**前提条件**（依赖哪些知识/安装）
- 可直接运行的**代码示例**
- 小节标题用动词短语（"注册事件处理器" 而非 "事件处理器"）
- 篇末**延伸阅读**链接（指向参考文档或下一篇指南）

**参考文档（reference/）**
- 类/函数签名（完整类型注解）
- 参数表格（参数名 | 类型 | 默认值 | 说明）
- 至少一个最简使用示例
- 异常/边界行为说明

**贡献文档（contributing/）**
- 内部实现细节，不需要对用户友好
- 可以包含 ADR（架构决策记录）
- 说明为什么这样设计，而不仅是怎么实现

### 2.4 代码块规范

- 始终标注语言标识符：` ```python ` / ` ```toml ` / ` ```bash `
- 示例代码必须**可以直接复制运行**（有完整 import）
- 输出/注释以 `# ...` 行内标注，不单独列出 stdout

## 三、写作风格

### 3.1 语言与语气

- 使用**简体中文**；技术术语（类名、方法名、参数名）保留英文原文。
- 语气：直接、简洁，避免废话（例如"我们将会介绍..."）。
- 句式偏短；一个知识点一个段落。

### 3.2 标题层级

- H1（`#`）：仅文件标题，每文件只有一个
- H2（`##`）：主要章节
- H3（`###`）：子节
- H4（`####`）：谨慎使用，避免超过 4 层嵌套

### 3.3 跨文档引用

- 使用相对路径链接，例如 `[参考文档](../reference/plugin/1_base_class.md)`
- 引用时在括号内附一句说明，例如 `（参见 [ConfigMixin](...)）`
- 禁止硬编码网站绝对路径（文档可能在多处部署）

### 3.4 禁止事项

- 不要在文档中列出 TODO / FIXME / 占位文本
- 不要复制粘贴大段源码
- 不要在一个文件中混合多个不相关主题

## 四、维护规范

### 4.1 新增文档时需同步更新的文件

| 操作 | 需要同步更新的文件 |
|------|-------------------|
| 在某目录下新增 `.md` 文件 | 该目录的 `README.md`（概览表 + 导航） |
| 新建子目录 | 父目录的 `README.md` + `docs/README.md` 目录树 |
| 重命名文件 | 所有引用该文件的文档（用 grep 搜索） |
| 删除文件 | 所有引用该文件的链接（改为链接到替代文档） |

### 4.2 修改已有文档时的原则

- **只修改需要修改的部分**；不做无关的格式调整或重构。
- 若修改导致文件超过 400 locs，先拆分，再修改。
- 修改 API / 行为描述后，检查 `reference/` 和 `guide/` 是否需要同步。

### 4.3 文档与代码同步

- 代码重构时，必须同步更新相关的 `reference/` 文档（签名、参数）。
- 新增公开 API 时，必须在 `reference/` 中添加对应条目，并酌情在 `guide/` 中提供用法示例。
- 删除/废弃 API 时，在原条目添加 `> **已废弃**` 提示，注明替代方案。

### 4.4 验证清单

在提交文档修改之前：

- [ ] 新文件已添加到父目录 `README.md` 的概览表
- [ ] `docs/README.md` 目录树已更新（如新增/删除文件）
- [ ] 所有内部链接可以正确解析（相对路径无断链）
- [ ] 文件 locs ≤ 400（用 `wc -l` 或编辑器行号确认）
- [ ] 代码示例语言标识符已标注
